Guía para principiantes sobre las políticas de ejecución en PowerShell (comandos y ejemplos seguros)



Las políticas de ejecución de PowerShell son reglas que determinan si un script (.ps1) puede ejecutarse y en qué condiciones.Aunque no son una medida de seguridad definitiva, sí actúan como una barrera preventiva para evitar que se ejecuten scripts no deseados o potencialmente maliciosos.



En esta guía aprenderás, paso a paso, cómo funcionan, qué tipos existen, y cómo gestionarlas de forma segura, sin comprometer tu sistema.

Qué son las políticas de ejecución



Las políticas de ejecución son un mecanismo interno de PowerShell que decide si un script puede o no ejecutarse. No reemplazan un antivirus ni una solución EDR: un usuario con permisos administrativos puede cambiarlas o ignorarlas.Sin embargo, son útiles para prevenir la ejecución accidental o automática de código no confiable.

Tipos de políticas de ejecución




Restricted



La más estricta: no permite ejecutar ningún script. Solo podemos usar comandos de forma interactiva en la consola. Es una opción muy segura, útil cuando no queremos ejecutar scripts en absoluto.



RemoteSigned



Permite ejecutar scripts locales sin firmar, pero exige firma para los scripts descargados de Internet (los que tienen “marca de procedencia”). Suele ser una buena recomendación para el día a día en Windows.



AllSigned



Solo se ejecutan scripts firmados digitalmente por un publicador de confianza (incluye los que creemos nosotros si están firmados). Es muy segura, pero requiere firmar todo lo que ejecutemos.



Unrestricted



Permite ejecutar cualquier script. En Windows, muestra una advertencia la primera vez que ejecutemos un script con marca de procedencia. Es menos segura, por lo que no la usaríamos como ajuste permanente.



Bypass



No bloquea ni muestra advertencias. La usamos solo de forma puntual para pruebas o automatizaciones controladas, nunca como configuración permanente del sistema.



Undefined



Significa “sin valor en este alcance”. PowerShell buscará la política en otro alcance. Si en todos los niveles está Undefined, aplica el valor predeterminado del sistema.




Alcances y prioridad



Las políticas pueden definirse en distintos alcances (scopes), con prioridades específicas:



AlcanceDescripciónPersistePrioridadMachinePolicyDefinida por GPO a nivel de equipoSíMáximaUserPolicyDefinida por GPO a nivel de usuarioSíAltaProcessSolo para la sesión actualNoMediaCurrentUserSolo afecta al usuario actualSíMedia-bajaLocalMachinePara todos los usuarios del equipoSíBaja



!! Debemos tener siempre en cuenta estas reglas prácticas !!




Si hay GPO (MachinePolicy o UserPolicy), siempre ganan sobre los demás alcances.



Entre valores locales, Process &gt; CurrentUser &gt; LocalMachine.



Si un alcance está en Undefined, se salta al siguiente.



CurrentUser prevalece sobre LocalMachine si ambos están definidos.




Comandos básicos para revisar la configuración



# Ver la política efectiva actual
Get-ExecutionPolicy

# Ver todas las políticas por alcance
Get-ExecutionPolicy -List

# Ver la versión de PowerShell
$PSVersionTable.PSVersion












Cómo cambiar la política de forma segura



Si estás aprendiendo o trabajando en un entorno personal, modifica solo tu usuario o la sesión actual, nunca el sistema completo.



# Cambiar solo para el usuario actual
Set-ExecutionPolicy RemoteSigned -Scope CurrentUser

# Cambio temporal (se borra al cerrar PowerShell)
Set-ExecutionPolicy Bypass -Scope Process

# Restablecer el valor en CurrentUser
Set-ExecutionPolicy Undefined -Scope CurrentUser
















Ejecuciones puntuales sin tocar la configuración



Para ejecutar un script puntual sin modificar la política global:



# PowerShell clásico
powershell.exe -ExecutionPolicy Bypass -File .\\script.ps1

# PowerShell 7+
pwsh -NoProfile -ExecutionPolicy Bypass -File .\\script.ps1



Ejecutar como administrador (si es necesario):



Start-Process -FilePath \"powershell.exe\" -Verb RunAs








En PowerShell 7, cambiamos \"powershell.exe\" por \"pwsh\".



Linux y macOS: en PowerShell 7+, la política de ejecución no se aplica; el parámetro -ExecutionPolicy no hace cumplir restricciones.



Importante:




Bypass/Unrestricted reducen protección: úsalas solo temporalmente (Process o con el parámetro -ExecutionPolicy al invocar).



Las políticas definidas por GPO residen bajo Policies y no pueden anularse localmente.



Las políticas impuestas por directivas de grupo (MachinePolicy y UserPolicy) tienen prioridad máxima y no podemos cambiarlas con Set-ExecutionPolicy.




Scripts descargados y marca de procedencia



    Los archivos descargados desde Internet incluyen una marca de procedencia (Mark of the Web).Si usamos RemoteSigned, PowerShell requerirá que estén firmados o desbloqueados.



# Desbloquear un script individual
Unblock-File -Path .\\script.ps1

# Desbloquear varios scripts (si confiamos en todos)
Get-ChildItem -Path . -Filter *.ps1 -Recurse | Unblock-File












Alternativa visual: clic derecho → Propiedades → “Desbloquear”.



Firma de scripts (más seguridad)



Firmar un script garantiza su integridad y autenticidad.Podemos crear un certificado local de firma de código para uso personal:



Crear un certificado de firma de código para uso personal (laboratorio) en Windows:



Creamos el certificado desde el directorio LocalMachine\\My



$cert = New-SelfSignedCertificate -Type CodeSigningCert -Subject \"CN=Firma Scripts Local\" `-CertStoreLocation Cert:\\LocalMachine\\My -KeyAlgorithm RSA -KeyLength 3072 `
-NotAfter (Get-Date).AddYears(2)











Vamos a detallar el proceso paso a paso:



# 1 Obtenemos la firma de nuestro script: (solo si el archivo ya cuenta con #certificado valido)

$sig = Get-AuthenticodeSignature .\\calc.ps1

# 2 Creamos la carpeta temporal (si no existe)

New-Item -ItemType Directory -Path C:\\Temp -Force | Out-Null

# 3 Definimos la ruta de nuestro .cer

$cerPath = 'C:\\Temp\\signer.cer'

# 4 Exportamos el certificado del firmante

Export-Certificate -Cert $sig.SignerCertificate -FilePath $cerPath

# 5 Comprobamos el valor de la variable

 $cerPath

# 6 Comprobamos que el archivo existe

Test-Path $cerPath

# 7 y 8 - Importamos a nivel equipo (privilegios de administrador)

Import-Certificate -FilePath $cerPath -CertStoreLocation Cert:\\LocalMachine\\Root

Import-Certificate -FilePath $cerPath -CertStoreLocation Cert:\\LocalMachine\\TrustedPublisher

# Si queremos solo para el usuario actual se importaría con los siguientes comandos

Import-Certificate -FilePath $cerPath -CertStoreLocation Cert:\\CurrentUser\\Root 

Import-Certificate -FilePath $cerPath -CertStoreLocation Cert:\\CurrentUser\\TrustedPublisher







    Luego, exportamos e importamos el certificado en los almacenes adecuados.Una vez instalado, podemos firmar el script:



# 1 - Revisamos los certificados disponibles

Get-ChildItem Cert:\\CurrentUser\\My -CodeSigningCert | Format-Table Subject,Thumbprint,NotAfter,HasPrivateKey

# 2 - Cargamos el que se encuentre en primer lugar (el mas actualizado)

$cert = Get-ChildItem Cert:\\CurrentUser\\My -CodeSigningCert | Where-Object HasPrivateKey | Sort-Object NotAfter -Descending | Select-Object -First 1

# 3 - Firmamos el script con el certificado elegido

Set-AuthenticodeSignature -FilePath .\\calc.ps1 -Certificate $cert -HashAlgorithm SHA256 -TimestampServer 'http://timestamp.digicert.com'

# 4 - Confirmamos que el script se encuentra firmado y con un certificado valido







    Para producción, lo ideal es usar un certificado emitido por una entidad de certificación confiable y distribuirlo a “Publicadores de confianza” en los equipos destino.



Buenas prácticas



✅ Usa RemoteSigned en CurrentUser como punto de partida.✅ Evita Unrestricted o Bypass permanentes.✅ Lee siempre el contenido de un script antes de ejecutarlo.✅ Mantén PowerShell y Windows actualizados.✅ Usa -NoProfile para probar scripts de terceros.✅ Control de versiones con Git y revisiones de código.✅ En entornos corporativos, aplica AllSigned con certificados oficiales.



Solución de problemas comunes



Error:




“Set-ExecutionPolicy no puede aplicar el cambio porque hay una política en un alcance más específico.”➡ Causa: GPO impuesta.➡ Solución: Revisar o contactar con el administrador de dominio.




En Linux/macOS:La política es Unrestricted, y no se aplica realmente.



Error al firmar scripts:Verifica que el certificado sea de tipo Code Signing y esté en TrustedPublisher.
